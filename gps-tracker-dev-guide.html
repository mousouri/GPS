Data LayerDatabase Design

üìå Location History Table

Core table: device_id, latitude, longitude, altitude, speed, heading, accuracy, recorded_at. Partitioned by time. Indexed on device_id + recorded_at.

TimescaleDB PostGIS GEOMETRY
‚ö° Live Position Cache

Store ONLY the latest position per device in Redis. Key: pos:{device_id}. Expire after inactivity. Read by the live map.

Redis Hash TTL expiry
üî∑ Geofence Storage

Store zones as PostGIS POLYGON or GEOMETRY types. Use ST_Contains() or ST_Intersects() for efficient server-side intersection queries.

ST_Contains GeoJSON
üöó Trip Detection

Auto-segment continuous location points into trips based on ignition events or idle gap thresholds. Store start/end, duration, distance per trip.

Batch Job Cron
üì¶ Data Retention Policy

Define how long raw location points are kept. Compress old data by downsampling (e.g., keep 1 point/minute after 90 days). Use TimescaleDB compression chunks.

TimescaleDB Compression
üë• Multi-Tenant Schema

Org ‚Üí Users ‚Üí Devices ‚Üí Assets hierarchy. Row-level security (RLS) in PostgreSQL to enforce tenant isolation at DB level.

RLS Policies Schema-per-tenant

CriticalSecurity & Privacy

‚ö† Legal Warning GPS tracking without informed consent is illegal in most jurisdictions. Always implement explicit consent flows and data processing agreements. Consult a lawyer for GDPR / CCPA compliance.
  • TLS/HTTPS for all connections ‚Äî no exceptions
  • MQTT over TLS (port 8883) for hardware trackers
  • JWT access tokens (short-lived) + refresh tokens
  • Device API keys for hardware auth (hashed in DB)
  • Rate limiting on ingestion endpoints
  • Role-Based Access Control (RBAC) ‚Äî enforce at API level
  • Row-Level Security in PostgreSQL per tenant
  • Encrypt location data at rest (AES-256)
  • GDPR: data export, deletion, consent tracking
  • Audit log for all data access events
  • Input validation and sanitization on all endpoints
  • CORS policy ‚Äî restrict allowed origins
  • Webhook signature verification
  • PII masking in logs
  • Geofence data is sensitive ‚Äî access-control it too
  • Allow users to see who can access their location

IoT IntegrationHardware Tracker Protocols

‚Ñπ Note Real GPS trackers don't use a browser ‚Äî they send raw TCP/UDP packets or use MQTT. Your backend must decode these proprietary protocols.
üì° MQTT

Lightweight pub/sub protocol. Ideal for battery-powered trackers. Device publishes to topic gps/{device_id}/location. Your server subscribes and writes to DB.

EMQXMosquitto
üîå TCP/UDP Raw Sockets

Cheap GPS modules (like Concox, Teltonika) send binary packets over TCP. You must implement a protocol parser for each device model.

Node.js net moduleGo TCP server
üåê HTTP Webhooks

Some trackers (like Traccar-compatible devices) POST NMEA sentences or JSON to your API. Simplest to implement but higher overhead.

REST POSTNMEA parsing
üõ†Ô∏è Traccar (Open Source)

Pre-built server that handles 200+ GPS device protocols. You can run it yourself and pipe data to your own DB, saving months of protocol work.

Self-hostedDocker

InfrastructureDevOps & Deployment

  • Containerize everything with Docker + Docker Compose
  • Kubernetes for production scale (pods per service)
  • CI/CD pipeline ‚Äî GitHub Actions or GitLab CI
  • Blue/green or rolling deployments (zero downtime)
  • Horizontal scaling for ingestion service (stateless)
  • Sticky sessions or Redis adapter for WebSocket scaling
  • CDN for frontend assets (Cloudflare, AWS CloudFront)
  • Map tile caching to reduce Mapbox/Google API costs
  • Database read replicas for analytics queries
  • Automated database backups + point-in-time recovery
  • Centralized logging (ELK stack or Loki + Grafana)
  • APM / tracing (Datadog, Sentry, OpenTelemetry)
  • Uptime monitoring with PagerDuty alerts
  • Load testing before launch (k6, Locust)
  • VPC + private subnets for DB and internal services
  • Secrets management (HashiCorp Vault or AWS Secrets Manager)
